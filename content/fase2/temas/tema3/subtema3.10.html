3.10: Organización profesional de proyectos Java en el sistema de archivos

La forma en que se organizan los proyectos Java en el sistema de archivos tiene un impacto directo y significativo en la productividad del desarrollo, la facilidad de navegación del código, la eficiencia de las herramientas de construcción, la claridad para colaboradores, y la mantenibilidad a largo plazo. Un proyecto bien organizado permite que cualquier desarrollador Java comprenda su estructura inmediatamente, localice archivos rápidamente, y comprenda las relaciones entre componentes sin necesidad de documentación extensa. En contraste, un proyecto desorganizado con archivos dispersos en ubicaciones arbitrarias y sin convenciones claras crea fricción constante que reduce productividad y aumenta errores.

Principios fundamentales de organización

1. Separación clara de concerns: código fuente separado de archivos compilados, tests separados de código de producción, recursos separados de código, documentación en ubicación dedicada, y scripts en directorio apropiado.

2. Convenciones sobre configuración: usar la estructura de directorios estándar que herramientas de construcción e IDEs reconocen automáticamente. Maven y Gradle establecen convenciones ampliamente adoptadas.

3. Jerarquía lógica reflejando arquitectura: la estructura de paquetes y directorios debe reflejar la organización lógica del código, facilitando comprensión de la arquitectura mediante inspección de la estructura de directorios.

4. Escalabilidad: la estructura debe funcionar tanto para proyectos pequeños como grandes, sin requerir reorganización mayor a medida que el proyecto crece.

5. Compatibilidad con herramientas: la estructura debe ser compatible con Maven, Gradle, IntelliJ, Eclipse, VS Code, Git, y herramientas de análisis de código.

Estructura estándar de proyecto Maven/Gradle

Maven estableció una estructura de directorios que se convirtió en estándar de facto para proyectos Java. Gradle adoptó la misma convención. Incluso proyectos que no usan ninguno de los dos frecuentemente siguen esta estructura por familiaridad.

  mi-proyecto/
├── src/
│   ├── main/
│   │   ├── java/              # Código fuente Java de producción
│   │   │   └── com/empresa/proyecto/
│   │   │       ├── Main.java
│   │   │       ├── modelo/
│   │   │       ├── servicio/
│   │   │       └── util/
│   │   └── resources/         # Recursos de producción
│   │       ├── application.properties
│   │       └── logback.xml
│   └── test/
│       ├── java/              # Código fuente de tests
│       │   └── com/empresa/proyecto/
│       └── resources/         # Recursos para tests
├── target/                    # Archivos compilados Maven (ignorado por Git)
├── build/                     # Archivos compilados Gradle (ignorado por Git)
├── docs/
├── scripts/
├── .gitignore
├── pom.xml
├── build.gradle
└── README.md

Desglose detallado de directorios

src/main/java/ — Código fuente de producción

Contiene todo el código Java que será compilado y desplegado en producción. Organización de paquetes recomendada:

  com.empresa.proyecto/
├── Main.java
├── config/
│   ├── DatabaseConfig.java
│   └── SecurityConfig.java
├── modelo/
│   ├── Usuario.java
│   └── Producto.java
├── repositorio/
│   ├── UsuarioRepository.java
│   └── ProductoRepository.java
├── servicio/
│   ├── UsuarioService.java
│   └── ProductoService.java
├── controlador/
│   ├── UsuarioController.java
│   └── ProductoController.java
├── dto/
├── excepcion/
└── util/

Nomenclatura de paquetes: todo en minúsculas, nombres descriptivos que reflejan la función. Organización por capa (modelo, servicio, repositorio) para proyectos pequeños; por característica/módulo para proyectos grandes.

src/main/resources/ — Recursos de producción

Contiene archivos no-Java necesarios en producción. Incluidos en el classpath y accesibles mediante getResourceAsStream().

  resources/
├── application.properties
├── application-dev.properties
├── application-prod.properties
├── logback.xml
├── db/migration/
│   ├── V1__Initial_schema.sql
│   └── V2__Add_users_table.sql
├── static/
│   ├── css/
│   ├── js/
│   └── images/
├── templates/
└── i18n/
    ├── messages_es.properties
    └── messages_en.properties

src/test/java/ — Código de tests

La estructura de paquetes debe espejear exactamente la de src/main/java, facilitando la localización de tests para cada clase.

  com.empresa.proyecto/
├── modelo/
│   ├── UsuarioTest.java
│   └── ProductoTest.java
├── servicio/
│   ├── UsuarioServiceTest.java
│   └── ProductoServiceTest.java
└── util/
    └── FechaUtilTest.java

Convenciones de nomenclatura: <ClaseTesteada>Test.java; métodos descriptivos como testCalcularDescuentoConPorcentajeValido() o deberiaLanzarExcepcionCuandoUsuarioNoExiste().

target/ y build/ — Archivos generados

Contiene archivos .class compilados, JARs/WARs, reportes de tests, y documentación generada. Este directorio nunca debe ser versionado en Git:

  # .gitignore
target/
build/
*.class

Archivos en raíz del proyecto

pom.xml / build.gradle: define dependencias, plugins, versión de Java objetivo, repositorios, y configuración de empaquetado.

README.md: descripción del proyecto, requisitos, instrucciones de instalación y ejecución, y cómo ejecutar tests:

  # Sistema de Gestión de Inventario

## Requisitos
- Java 17 o superior
- Maven 3.8+
- PostgreSQL 14+

## Instalación
git clone https://github.com/empresa/inventario.git
cd inventario
mvn clean install

## Ejecución
mvn spring-boot:run
# Aplicación en: http://localhost:8080

## Tests
mvn test

Organización de proyectos multi-módulo

Para aplicaciones grandes, la estructura de proyecto único puede ser insuficiente. Los proyectos multi-módulo dividen la aplicación en módulos separados con dependencias explícitas.

  empresa-erp/
├── pom.xml                # POM padre
├── erp-modelo/            # Entidades de dominio
│   ├── pom.xml
│   └── src/main/java/
├── erp-repositorio/       # Acceso a datos
│   ├── pom.xml
│   └── src/main/java/
├── erp-servicio/          # Lógica de negocio
│   ├── pom.xml
│   └── src/main/java/
├── erp-api/               # API REST
│   ├── pom.xml
│   └── src/main/java/
└── erp-web/               # Interfaz web
    ├── pom.xml
    └── src/main/java/

Ventajas: separación de responsabilidades más clara, compilación independiente de módulos, reutilización, y aplicación de principios SOLID a nivel de módulos. Desventajas: mayor complejidad inicial; puede ser overkill para proyectos pequeños.

Ubicación del workspace de desarrollo

macOS/Linux — recomendado:

  ~/projects/
├── empresa/
│   ├── proyecto-a/
│   └── proyecto-b/
├── personal/
└── open-source/

Windows — recomendado:

  C:\dev\
├── empresa\
├── personal\
└── open-source\

Evitar: Escritorio, Documentos, Descargas, y rutas con espacios o caracteres especiales, ya que pueden causar problemas con herramientas de construcción.

Buenas prácticas adicionales

Un proyecto = un repositorio Git. Evitar monorepos accidentales donde múltiples proyectos no relacionados comparten el mismo repositorio.

Documentar estructura en README si difiere del estándar Maven/Gradle.

Mantener la raíz del proyecto limpia. Evitar acumular archivos sueltos sin directorio claro.

Scripts de utilidad en scripts/ o tools/.

Archivos .env en .gitignore. Proporcionar .env.example versionado con valores de ejemplo, nunca credenciales reales.

Consistencia entre proyectos. Una estructura uniforme facilita que los desarrolladores sean productivos inmediatamente al unirse a cualquier proyecto de la organización.

La organización profesional del sistema de archivos no es un detalle cosmético sino una práctica fundamental que afecta eficiencia, mantenibilidad, y escalabilidad. Adoptar convenciones establecidas y mantener separación clara de concerns facilita colaboración y permite que los desarrolladores sean productivos inmediatamente al unirse a un proyecto.

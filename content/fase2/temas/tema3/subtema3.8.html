3.8: Gestión de múltiples versiones de Java

En el desarrollo profesional de Java, es extremadamente común necesitar trabajar con múltiples versiones del JDK simultáneamente. Un desarrollador puede estar manteniendo un proyecto legacy que requiere Java 8, desarrollando una aplicación nueva con Java 21, y experimentando con características preview en la versión más reciente. Intentar gestionar esto reinstalando constantemente versiones diferentes o modificando manualmente variables de entorno cada vez es ineficiente, propenso a errores, y frustrante.

¿Por qué múltiples versiones?

1. Mantenimiento de proyectos legacy: Un proyecto antiguo escrito para Java 8 puede no actualizarse fácilmente, requiriendo esa versión para compilar y ejecutar mientras se trabaja en un proyecto nuevo con Java 17 o 21.

2. Compatibilidad con diferentes clientes/proyectos: Un cliente puede requerir Java 11 por política corporativa, otro permitir Java 17, y un proyecto personal usar Java 21.

3. Testing de compatibilidad: Verificar que una biblioteca funciona en Java 11, 17, y 21, o replicar localmente lo que CI/CD ejecuta en múltiples versiones.

4. Exploración y aprendizaje: Aprender características nuevas en versiones recientes mientras se mantiene una versión estable para trabajo de producción.

5. Requisitos de frameworks: Frameworks diferentes pueden requerir versiones mínimas de Java distintas entre sí.

Conceptos fundamentales

El JDK está diseñado para soportar múltiples versiones instaladas simultáneamente, cada una en su propio directorio completamente separado y auto-contenido.

Estructura típica con múltiples versiones en Windows:

  C:\Program Files\Eclipse Adoptium\
├── jdk-8.0.392.8-hotspot\
├── jdk-11.0.21.9-hotspot\
├── jdk-17.0.9.9-hotspot\
└── jdk-21.0.1.12-hotspot\

En macOS:

  /Library/Java/JavaVirtualMachines/
├── temurin-8.jdk/
├── temurin-11.jdk/
├── temurin-17.jdk/
└── temurin-21.jdk/

En Linux:

  /opt/
├── jdk-8/
├── jdk-11/
├── jdk-17/
└── jdk-21/

Cambiar la versión activa requiere modificar JAVA_HOME y PATH para apuntar a la versión deseada. Existen tres enfoques: edición manual (tedioso, no recomendado), scripts/aliases, y herramientas de gestión de versiones.

Método 1: Scripts y aliases (solución simple)

Para bash/zsh (macOS/Linux) — agregar a ~/.bashrc o ~/.zshrc:

  # ===== Gestión de Versiones Java =====

java8() {
    export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-8.jdk/Contents/Home
    export PATH=$JAVA_HOME/bin:$PATH
    echo "Cambiado a Java 8"
    java -version
}

java11() {
    export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-11.jdk/Contents/Home
    export PATH=$JAVA_HOME/bin:$PATH
    echo "Cambiado a Java 11"
    java -version
}

java17() {
    export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-17.jdk/Contents/Home
    export PATH=$JAVA_HOME/bin:$PATH
    echo "Cambiado a Java 17"
    java -version
}

java21() {
    export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-21.jdk/Contents/Home
    export PATH=$JAVA_HOME/bin:$PATH
    echo "Cambiado a Java 21"
    java -version
}

# Establecer Java 21 como default
java21

Uso:

  java11          # Cambiar a Java 11
java -version   # Verificar versión activa
java21          # Cambiar a Java 21

source ~/.bashrc  # Aplicar cambios

Para macOS usando /usr/libexec/java_home (método recomendado en macOS):

  # Función que usa java_home de macOS
setjdk() {
    if [ $# -eq 0 ]; then
        echo "Uso: setjdk "
        echo "Ejemplo: setjdk 21"
        /usr/libexec/java_home -V
        return 1
    fi
    export JAVA_HOME=$(/usr/libexec/java_home -v $1)
    export PATH=$JAVA_HOME/bin:$PATH
    echo "JAVA_HOME establecido a Java $1"
    java -version
}

# Default a Java 21
setjdk 21

  setjdk 11   # Cambiar a Java 11
setjdk 17   # Cambiar a Java 17
/usr/libexec/java_home -V   # Ver versiones disponibles

Para Windows (PowerShell) — agregar al perfil $PROFILE:

  # Abrir perfil en editor
notepad $PROFILE
# Si no existe: New-Item -Path $PROFILE -Type File -Force

  # ===== Gestión de Versiones Java =====

function Set-Java8 {
    $env:JAVA_HOME = "C:\Program Files\Eclipse Adoptium\jdk-8.0.392.8-hotspot"
    $env:PATH = "$env:JAVA_HOME\bin;$env:PATH"
    Write-Host "Cambiado a Java 8" -ForegroundColor Green
    java -version
}

function Set-Java17 {
    $env:JAVA_HOME = "C:\Program Files\Eclipse Adoptium\jdk-17.0.9.9-hotspot"
    $env:PATH = "$env:JAVA_HOME\bin;$env:PATH"
    Write-Host "Cambiado a Java 17" -ForegroundColor Green
    java -version
}

function Set-Java21 {
    $env:JAVA_HOME = "C:\Program Files\Eclipse Adoptium\jdk-21.0.1.12-hotspot"
    $env:PATH = "$env:JAVA_HOME\bin;$env:PATH"
    Write-Host "Cambiado a Java 21" -ForegroundColor Green
    java -version
}

Set-Alias java17 Set-Java17
Set-Alias java21 Set-Java21

# Default a Java 21
Set-Java21

  . $PROFILE   # Recargar perfil
java21       # Cambiar a Java 21
java17       # Cambiar a Java 17

Ventajas: simple, sin software adicional, control total. Desventajas: cambios solo afectan la sesión actual, no integra con proyectos, requiere configuración manual inicial.

Método 2: SDKMAN (recomendado para macOS/Linux)

SDKMAN gestiona versiones de múltiples SDKs: JDK, Gradle, Maven, Kotlin, Scala, y más.

Instalación:

  curl -s "https://get.sdkman.io" | bash
source "$HOME/.sdkman/bin/sdkman-init.sh"

sdk version   # Verificar instalación

Uso básico:

  # Ver versiones disponibles
sdk list java

# Instalar versiones
sdk install java 21.0.1-tem
sdk install java 17.0.9-tem
sdk install java 11.0.21-tem
sdk install java 21.0.1-amzn   # Amazon Corretto

# Ver versiones instaladas
sdk list java | grep installed

# Cambiar versión para sesión actual
sdk use java 11.0.21-tem

# Establecer default global
sdk default java 21.0.1-tem

# Verificar versión activa
sdk current java

# Desinstalar
sdk uninstall java 11.0.21-tem

# Actualizar SDKMAN
sdk update
sdk upgrade

Configuración por proyecto con .sdkmanrc:

  # Crear archivo en raíz del proyecto
echo "java=17.0.9-tem" > .sdkmanrc

# Al entrar al directorio del proyecto
sdk env install   # Instala versión si no está instalada
sdk env           # Activa la versión especificada

Para activación automática al entrar al directorio, agregar a ~/.bashrc o ~/.zshrc:

  sdkman_auto_env=true

Ventajas: instalación y gestión de versiones muy simple, gestiona múltiples SDKs, soporte para configuración por proyecto. Desventajas: solo macOS/Linux (no Windows nativo, requiere WSL), instalaciones en ubicación fija (~/.sdkman).

Método 3: jEnv (alternativa específica para Java)

jEnv gestiona únicamente versiones de Java, con énfasis en configuración por directorio.

Instalación:

  # macOS con Homebrew
brew install jenv

# Linux (manual)
git clone https://github.com/jenv/jenv.git ~/.jenv

# Configurar shell (bash)
echo 'export PATH="$HOME/.jenv/bin:$PATH"' >> ~/.bashrc
echo 'eval "$(jenv init -)"' >> ~/.bashrc
source ~/.bashrc

# Configurar shell (zsh)
echo 'export PATH="$HOME/.jenv/bin:$PATH"' >> ~/.zshrc
echo 'eval "$(jenv init -)"' >> ~/.zshrc
source ~/.zshrc

Uso básico:

  # Agregar versiones ya instaladas a jEnv
jenv add /Library/Java/JavaVirtualMachines/temurin-11.jdk/Contents/Home
jenv add /Library/Java/JavaVirtualMachines/temurin-17.jdk/Contents/Home
jenv add /Library/Java/JavaVirtualMachines/temurin-21.jdk/Contents/Home

# Ver versiones disponibles
jenv versions

# Cambiar versión global
jenv global 21.0

# Cambiar versión para directorio actual (crea .java-version)
cd ~/proyectos/proyecto-legacy
jenv local 11.0

# Cambiar versión solo para sesión actual
jenv shell 17.0

# Verificar versión activa
jenv version
java -version

Habilitar plugins útiles:

  jenv enable-plugin export   # Configura JAVA_HOME automáticamente
jenv enable-plugin maven
jenv enable-plugin gradle

Ventajas: específico para Java, configuración por directorio con .java-version, plugins para herramientas de construcción. Desventajas: no descarga versiones automáticamente, no gestiona otros SDKs, requiere agregar cada versión manualmente.

Método 4: Jabba (multiplataforma incluyendo Windows)

Instalación:

  # macOS/Linux
curl -sL https://github.com/shyiko/jabba/raw/master/install.sh | bash && . ~/.jabba/jabba.sh

  # Windows (PowerShell como Administrador)
[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12
Invoke-Expression (Invoke-WebRequest https://github.com/shyiko/jabba/raw/master/install.ps1 -UseBasicParsing).Content

  # Listar versiones disponibles
jabba ls-remote

# Instalar y usar
jabba install adopt@1.21.0-1
jabba use adopt@1.21.0-1
jabba alias default adopt@1.21.0-1

Recomendaciones por caso de uso

Desarrollo individual en macOS/Linux: SDKMAN (más completo, gestiona también Maven/Gradle) o jEnv (si solo se necesita gestión de Java).

Desarrollo individual en Windows: scripts PowerShell (simple), Jabba (más sofisticado), o SDKMAN en WSL.

Para equipos: estandarizar en una herramienta común (típicamente SDKMAN), incluir archivo de configuración de proyecto (.sdkmanrc o .java-version), y documentar en el README.

Para principiantes: empezar con scripts simples y migrar a una herramienta dedicada cuando la necesidad sea clara.

Buenas prácticas

1. Documentar versión requerida en el README del proyecto:

  ## Requisitos
- Java 17 o superior
- Recomendado: Eclipse Temurin 17.0.9

2. Incluir archivo de configuración de versión (.sdkmanrc, .java-version, o .tool-versions) en el repositorio.

3. Especificar versión en el sistema de construcción:

  

  17
  17


  // build.gradle (Gradle)
java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(17)
    }
}

4. Configurar CI/CD con matrices de versiones:

  # GitHub Actions
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        java: [11, 17, 21]
    steps:
      - uses: actions/setup-java@v3
        with:
          java-version: ${{ matrix.java }}

5. Mantener las versiones instaladas actualizadas: aplicar CPUs de seguridad regularmente, eliminar versiones muy antiguas ya no usadas, y mantener al menos una versión LTS reciente.

La capacidad de gestionar múltiples versiones de Java eficientemente es habilidad esencial para desarrolladores profesionales modernos. Ya sea mediante scripts simples o herramientas sofisticadas como SDKMAN, tener un método establecido para cambiar entre versiones elimina fricción, previene errores, y permite trabajar productivamente en múltiples proyectos con requisitos diferentes.

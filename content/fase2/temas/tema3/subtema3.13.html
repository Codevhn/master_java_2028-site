3.13: Errores comunes durante la instalación y configuración

A pesar de que la instalación y configuración del entorno Java es relativamente directa cuando se siguen los pasos correctamente, numerosos errores comunes surgen repetidamente, especialmente para desarrolladores que configuran Java por primera vez o que trabajan en sistemas con configuraciones previas conflictivas. Estos errores frecuentemente causan frustración porque los mensajes de error pueden ser crípticos, las causas no son inmediatamente obvias, y las soluciones requieren comprender detalles de cómo el sistema operativo, variables de entorno, y herramientas Java interactúan. Este subtema cataloga los errores más comunes, explica sus causas raíz, y proporciona soluciones paso a paso.

Categoría 1: Errores de comando no encontrado

Error 1.1: "'java' is not recognized as an internal or external command"

Plataforma: Windows

  C:\Users\usuario> java -version
'java' is not recognized as an internal or external command,
operable program or batch file.

Causas posibles:

  1. JDK no está instalado
  2. PATH no está configurado correctamente
  3. PATH está configurado pero terminal no ha sido reiniciada
  4. Instalación de JDK está corrupta

Solución paso a paso:

Verificar si JDK está instalado:

  dir "C:\Program Files\Eclipse Adoptium"
dir "C:\Program Files\Java"

Si no muestra ningún directorio → instalar JDK (ver subtema 3.5).

Si JDK está instalado, verificar PATH:

  echo %PATH%

Buscar en la salida referencias a Java. Debe contener algo como:

  ...;C:\Program Files\Eclipse Adoptium\jdk-21.0.1.12-hotspot\bin;...

Si PATH no contiene ruta a JDK, seguir instrucciones de configuración en subtema 3.6. Importante: después de configurar, cerrar y abrir nueva ventana de CMD.

Si PATH parece correcto pero el comando sigue fallando:

  # Verificar JAVA_HOME
echo %JAVA_HOME%

# Verificar manualmente que java.exe existe
dir "%JAVA_HOME%\bin\java.exe"

Error 1.2: "bash: java: command not found"

Plataforma: macOS / Linux

  $ java -version
bash: java: command not found

Causas posibles: JDK no está instalado, PATH no está configurado, archivo de configuración de shell incorrecto, o cambios no aplicados a sesión actual.

Verificar si JDK está instalado:

  # macOS
ls /Library/Java/JavaVirtualMachines/

# Linux
ls /opt/ | grep jdk
ls /usr/lib/jvm/

Verificar JAVA_HOME y PATH:

  echo $JAVA_HOME
echo $PATH | tr ':' '\n' | grep java

Si JAVA_HOME está vacío, verificar qué shell está activo:

  echo $SHELL

Para bash:

  nano ~/.bashrc   # Linux
nano ~/.bash_profile  # macOS

# Agregar al final:
export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-21.jdk/Contents/Home
export PATH=$JAVA_HOME/bin:$PATH

# Aplicar cambios
source ~/.bashrc  # o source ~/.bash_profile

Para zsh:

  nano ~/.zshrc

# Agregar mismo contenido que bash
export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-21.jdk/Contents/Home
export PATH=$JAVA_HOME/bin:$PATH

# Guardar y aplicar
source ~/.zshrc

Categoría 2: Errores de versión incorrecta

Error 2.1: Versión de Java diferente a la esperada

  $ java -version
openjdk version "1.8.0_392"
# Pero se instaló Java 21

Causa: múltiples instalaciones de Java en el sistema, otra versión está primero en PATH.

Identificar todas las instalaciones:

  # macOS
/usr/libexec/java_home -V

# Linux
update-alternatives --list java   # Debian/Ubuntu
alternatives --list java          # RHEL/Fedora

# Windows
where java

Asegurar que JAVA_HOME/bin está al inicio del PATH:

  export PATH=$JAVA_HOME/bin:$PATH   # ✓ Correcto

export PATH=$PATH:$JAVA_HOME/bin   # ✗ Incorrecto: pone al final

En Windows, usar el editor de variable PATH y mover la entrada del JDK deseado al tope de la lista.

Error 2.2: javac y java muestran versiones diferentes

  $ javac -version
javac 21.0.1

$ java -version
openjdk version "11.0.21"

Causa: javac y java resolviendo a diferentes instalaciones.

  which javac
which java

Si muestran rutas diferentes, PATH contiene referencias a múltiples instalaciones. Limpiar PATH para incluir solo una versión:

  export PATH=$JAVA_HOME/bin:$PATH

java -version
javac -version   # Ambos deben mostrar misma versión

Categoría 3: Errores de compilación

Error 3.1: "javac: command not found" pero java funciona

  $ java -version
openjdk version "21.0.1"   # ✓ Funciona

$ javac -version
bash: javac: command not found   # ✗ Falla

Causa: se instaló JRE en lugar de JDK.

  ls $JAVA_HOME/bin/

El JDK completo incluye: java, javac, jar, javadoc, etc. Si solo tiene java pero no javac, hay que desinstalar el JRE e instalar el JDK completo (ver subtema 3.5).

Error 3.2: "error: release version X not supported"

  $ javac -release 21 MiClase.java
error: release version 21 not supported

Causa: versión de javac es más antigua que la versión especificada con -release. Solución: actualizar al JDK que soporte la versión deseada, o ajustar -release a la versión soportada por el JDK instalado.

Error 3.3: "error: cannot find symbol"

  $ javac MiClase.java
MiClase.java:5: error: cannot find symbol
String nombre = "Java";
^
symbol:   class String
location: class MiClase

La causa más común es un error de sintaxis en el código (typo, importación faltante). Verificar el código cuidadosamente. Para confirmar que la instalación no está corrupta:

  ls $JAVA_HOME/jmods/   # Java 9+
ls $JAVA_HOME/lib/     # Java 8

Si los directorios están vacíos, reinstalar el JDK.

Categoría 4: Errores de ejecución

Error 4.1: "Error: Could not find or load main class"

  $ java MiClase
Error: Could not find or load main class MiClase
Caused by: java.lang.ClassNotFoundException: MiClase

Causa A — Nombre incorrecto: no incluir la extensión .class:

  java MiClase.class   # ✗ Incorrecto
java MiClase         # ✓ Correcto

Causa B — Clase en paquete: si MiClase.java declara un paquete, usar el nombre completo:

  package com.empresa.app;

public class MiClase {
    public static void main(String[] args) {
        System.out.println("Hola");
    }
}

  java MiClase                    # ✗ Ignora el paquete
java com.empresa.app.MiClase    # ✓ Nombre completo con paquete

Causa C — Directorio incorrecto:

  # Si el .class está en proyecto/src/MiClase.class
# y ejecutas desde proyecto/:
java MiClase          # ✗ No encuentra la clase

# Solución: especificar classpath
java -cp src MiClase  # ✓

Error 4.2: "UnsupportedClassVersionError"

  $ java MiClase
Exception in thread "main" java.lang.UnsupportedClassVersionError:
MiClase has been compiled by a more recent version of the Java Runtime
(class file version 65.0), this version of the Java Runtime only
recognizes class file versions up to 61.0

Versiones de class file: 65.0 = Java 21, 61.0 = Java 17, 55.0 = Java 11, 52.0 = Java 8.

Causa: clase compilada con versión más nueva de Java que la JVM que la ejecuta.

Opción 1 — usar la misma versión para ejecutar:

  java21         # Cambiar a Java 21
java -version  # Verificar
java MiClase

Opción 2 — compilar para versión más antigua:

  javac --release 17 MiClase.java   # Compilar compatible con Java 17
java MiClase                      # Ejecutar con Java 17

Categoría 5: Errores de JAVA_HOME

Error 5.1: JAVA_HOME apunta a ubicación incorrecta

  echo $JAVA_HOME
ls $JAVA_HOME   # Debe mostrar: bin, lib, include, etc.

Si muestra "No such file or directory", JAVA_HOME está mal configurado.

Error frecuente en macOS — falta /Contents/Home:

  # Incorrecto:
export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-21.jdk

# Correcto:
export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-21.jdk/Contents/Home

Error frecuente en Windows — apuntando al subdirectorio bin:

  # Incorrecto:
JAVA_HOME=C:\Program Files\Eclipse Adoptium\jdk-21\bin

# Correcto:
JAVA_HOME=C:\Program Files\Eclipse Adoptium\jdk-21

Error 5.2: JAVA_HOME no persiste después de reiniciar

Causa: variable configurada solo para sesión actual, no de forma permanente.

En Windows, usar la configuración de variables de entorno del sistema (no set en CMD). En macOS/Linux, agregar el export al archivo de shell correcto (~/.bashrc, ~/.bash_profile, o ~/.zshrc).

Verificar que el archivo se carga al inicio:

  # Agregar línea de test temporal
echo "Cargando configuración de shell" >> ~/.zshrc

# Abrir nueva terminal — debe mostrar el mensaje

Categoría 6: Errores de permisos

Error 6.1: "Permission denied" al instalar JDK (macOS/Linux)

  sudo mv jdk-21 /opt/   # Usar sudo para directorios del sistema

Alternativa sin sudo — instalar en directorio de usuario:

  mkdir -p ~/java
mv jdk-21 ~/java/
export JAVA_HOME=~/java/jdk-21

Error 6.2: "Access is denied" al configurar variables de entorno (Windows)

Si la cuenta no tiene privilegios administrativos, usar la sección de variables de entorno de usuario (no de sistema) en el diálogo — no requiere privilegios admin.

Categoría 7: Errores de IDEs

Error 7.1: IDE no encuentra JDK

IntelliJ IDEA: File → Project Structure → Project → Project SDK → Add SDK → JDK → navegar a la ruta del JDK instalado.

Eclipse: Window → Preferences → Java → Installed JREs → Add → Standard VM → navegar a JDK → Finish.

VS Code: instalar Extension Pack for Java → Command Palette (Ctrl+Shift+P) → "Java: Configure Java Runtime" → agregar JDK manualmente.

Error 7.2: Proyecto no compila en IDE pero funciona en línea de comandos

Causa: IDE usando configuración diferente de JDK que la línea de comandos.

Verificar que la configuración del proyecto apunte al mismo JDK que JAVA_HOME, y que el nivel de lenguaje del proyecto coincida con la versión usada (en IntelliJ: File → Project Structure → Project language level; en Eclipse: Project properties → Java Compiler → Compiler compliance level).

Checklist de troubleshooting general

Cuando enfrentes un error no cubierto arriba, seguir este proceso:

  # 1. Verificar instalación
ls $JAVA_HOME        # macOS/Linux
dir %JAVA_HOME%      # Windows

# 2. Verificar versión
java -version
javac -version

# 3. Verificar ubicación de ejecutables
which java           # macOS/Linux
which javac
where java           # Windows
where javac

# 4. Verificar variables de entorno
echo $JAVA_HOME
echo $PATH
  • Abrir nueva terminal/CMD — muchos problemas se resuelven con una sesión fresca.
  • Verificar que el archivo compiló: debe existir el .class correspondiente.
  • Verificar sintaxis del comando: java NombreClase — sin .class, sin .java.

La mayoría de errores de instalación y configuración caen en estas categorías. Comprender las causas raíz, no solo memorizar soluciones, permite diagnosticar y resolver problemas eficientemente cuando surgen variaciones de estos errores en contextos específicos.

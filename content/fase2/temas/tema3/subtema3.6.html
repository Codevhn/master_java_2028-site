3.6: Configuración de variables de entorno (JAVA_HOME y PATH)

Después de instalar el JDK, el sistema operativo aún no sabe dónde encontrarlo para usarlo. Las variables de entorno JAVA_HOME y PATH son el mecanismo mediante el cual se comunica al sistema operativo, a las herramientas de desarrollo, y a los IDEs dónde está ubicado el JDK y cómo acceder a sus comandos. Configurar estas variables correctamente es absolutamente crítico; sin esta configuración, comandos como java y javac no funcionarán desde línea de comandos, los sistemas de construcción como Maven y Gradle fallarán, y los IDEs tendrán problemas localizando el JDK.

¿Qué son las variables de entorno?

Las variables de entorno son pares nombre-valor que el sistema operativo mantiene y que están disponibles para todos los procesos ejecutándose en el sistema. Los programas pueden leer estas variables para obtener información de configuración sin que necesite ser especificada cada vez que el programa ejecuta.

Ejemplos de variables de entorno comunes:

  • PATH: Lista de directorios donde el sistema busca ejecutables
  • HOME (Unix) / USERPROFILE (Windows): Directorio home del usuario
  • TEMP / TMP: Directorio para archivos temporales

JAVA_HOME: ¿Qué es y por qué es necesaria?

JAVA_HOME es una variable de entorno que apunta al directorio raíz de la instalación del JDK (no al subdirectorio bin, sino a la raíz).

Ejemplos de valor correcto:

  • Windows: C:\Program Files\Eclipse Adoptium\jdk-21.0.1.12-hotspot
  • macOS: /Library/Java/JavaVirtualMachines/temurin-21.jdk/Contents/Home
  • Linux: /opt/jdk-21 o /usr/lib/jvm/temurin-21-jdk-amd64

¿Por qué es necesaria? Muchas herramientas Java dependen de JAVA_HOME para localizar el JDK:

  • Maven y Gradle: Usan JAVA_HOME para encontrar el JDK al construir proyectos
  • IDEs: IntelliJ IDEA y Eclipse frecuentemente usan JAVA_HOME como ubicación por defecto
  • Servidores de aplicaciones: Tomcat, Jetty, WildFly requieren JAVA_HOME
  • Scripts de deployment: Frecuentemente referencian JAVA_HOME
  • Herramientas de terceros: Muchas herramientas Java asumen que JAVA_HOME está configurada

Error común si no está configurada:

  ERROR: JAVA_HOME is not set and no 'java' command could be found in your PATH.

PATH: ¿Qué es y por qué agregar el JDK?

PATH es una variable de entorno que contiene lista de directorios separados por delimitador (; en Windows, : en Unix) donde el sistema operativo busca ejecutables cuando se escribe un comando. Si $JAVA_HOME/bin (Unix) o %JAVA_HOME%\bin (Windows) no está en PATH, el sistema no encontrará los comandos java y javac.

Error común si no está en PATH:

  java -version
# bash: java: command not found

javac HelloWorld.java
# 'javac' is not recognized as an internal or external command

Configuración en Windows

Método 1: Interfaz gráfica (recomendado para principiantes)

Abrir Variables de Entorno en Windows 10/11:

  1. Click derecho en "This PC" → "Properties"
  2. Click "Advanced system settings"
  3. En la pestaña "Advanced", click "Environment Variables..."

Método alternativo (más rápido): Presionar Win + R, escribir sysdm.cpl, Enter → Pestaña "Advanced" → "Environment Variables..."

Crear JAVA_HOME:

  1. En "System variables", click "New..."
  2. Variable name: JAVA_HOME / Variable value: ruta completa al JDK sin \bin al final
  C:\Program Files\Eclipse Adoptium\jdk-21.0.1.12-hotspot
  1. Click "OK". La ruta debe apuntar al directorio que contiene bin, lib, include.

Modificar PATH:

  1. En "System variables", buscar Path y click "Edit..."
  2. Click "New" y agregar: %JAVA_HOME%\bin
  3. Click "OK" en todas las ventanas para guardar.

Método 2: PowerShell como Administrador

  # Establecer JAVA_HOME permanentemente
[System.Environment]::SetEnvironmentVariable('JAVA_HOME', 'C:\Program Files\Eclipse Adoptium\jdk-21.0.1.12-hotspot', 'Machine')

# Agregar al PATH
$currentPath = [System.Environment]::GetEnvironmentVariable('Path', 'Machine')
if ($currentPath -notlike "*%JAVA_HOME%\bin*") {
    $newPath = $currentPath + ";%JAVA_HOME%\bin"
    [System.Environment]::SetEnvironmentVariable('Path', $newPath, 'Machine')
}

Reiniciar PowerShell después de ejecutar estos comandos.

Método 3: CMD como Administrador

  setx JAVA_HOME "C:\Program Files\Eclipse Adoptium\jdk-21.0.1.12-hotspot" /M
setx PATH "%PATH%;%JAVA_HOME%\bin" /M

La opción /M establece la variable a nivel de sistema (requiere privilegios de administrador). Reiniciar CMD después de ejecutar estos comandos.

Configuración en macOS

La configuración depende del shell usado. macOS Catalina (10.15) y posteriores usan zsh por defecto; versiones anteriores usaban bash.

  # Determinar shell actual
echo $SHELL

Para zsh (macOS Catalina 10.15+)

Paso 1: Identificar ruta del JDK:

  /usr/libexec/java_home -V

Ejemplo de salida:

  Matching Java Virtual Machines (1):
    21.0.1 (x86_64) "Eclipse Adoptium" - "Eclipse Temurin 21" /Library/Java/JavaVirtualMachines/temurin-21.jdk/Contents/Home

Paso 2: Editar ~/.zshrc y agregar las variables:

  # Abrir archivo
nano ~/.zshrc

# Agregar al final:
# Java Environment
export JAVA_HOME=$(/usr/libexec/java_home -v 21)
export PATH=$JAVA_HOME/bin:$PATH

Alternativa con ruta explícita:

  export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-21.jdk/Contents/Home
export PATH=$JAVA_HOME/bin:$PATH

Paso 3: Guardar y aplicar cambios:

  # En nano: Ctrl+O (guardar), Enter, Ctrl+X (salir)

# Aplicar cambios inmediatamente
source ~/.zshrc

Para bash (macOS anterior a Catalina)

  # Editar archivo
nano ~/.bash_profile

# Agregar al final:
export JAVA_HOME=$(/usr/libexec/java_home -v 21)
export PATH=$JAVA_HOME/bin:$PATH

# Aplicar cambios
source ~/.bash_profile

Configuración en Linux

Para bash (más común)

Paso 1: Verificar ruta del JDK según método de instalación:

  • Paquete sistema: /usr/lib/jvm/temurin-21-jdk-amd64/
  • Manual en /opt: /opt/jdk-21/
  • Manual en home: ~/java/jdk-21/
  # Verificar contenido del directorio
ls /opt/jdk-21/
# Debe mostrar: bin conf include jmods legal lib release

Paso 2: Editar archivo de configuración:

  # Para usuario actual
nano ~/.bashrc

# Para todos los usuarios (requiere sudo)
sudo nano /etc/profile

Paso 3: Agregar variables al final del archivo:

  # Java Environment
export JAVA_HOME=/opt/jdk-21
export PATH=$JAVA_HOME/bin:$PATH

Paso 4: Guardar y aplicar:

  source ~/.bashrc
# o si editó /etc/profile:
source /etc/profile

Para zsh en Linux

  nano ~/.zshrc

# Agregar:
export JAVA_HOME=/opt/jdk-21
export PATH=$JAVA_HOME/bin:$PATH

source ~/.zshrc

Método alternativo: /etc/environment (Ubuntu/Debian)

  sudo nano /etc/environment

# Agregar:
JAVA_HOME="/opt/jdk-21"

# Modificar línea PATH existente:
PATH="/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:$JAVA_HOME/bin"

Reiniciar sesión para aplicar cambios.

Verificación de configuración correcta

Abrir una nueva ventana de terminal (importante: las ventanas abiertas antes de configurar no tendrán las variables nuevas) y ejecutar:

Verificar JAVA_HOME en Windows (CMD):

  echo %JAVA_HOME%

Verificar JAVA_HOME en Windows (PowerShell):

  echo $env:JAVA_HOME

Verificar JAVA_HOME en macOS/Linux:

  echo $JAVA_HOME

Debe mostrar la ruta al directorio JDK. Si muestra vacío, la variable no está configurada correctamente.

Verificar que java y javac funcionan:

  java -version

  openjdk version "21.0.1" 2023-10-17 LTS
OpenJDK Runtime Environment Temurin-21.0.1+12 (build 21.0.1+12-LTS)
OpenJDK 64-Bit Server VM Temurin-21.0.1+12 (build 21.0.1+12-LTS, mixed mode, sharing)

  javac -version

  javac 21.0.1

Verificar ubicación de los ejecutables:

  # macOS/Linux
which java
which javac

# Windows
where java
where javac

La salida debe mostrar una ruta dentro del directorio JDK instalado, no otra ubicación del sistema.

Problemas comunes y soluciones

Problema 1: "java/javac no se reconoce como comando". Causa: PATH no contiene el bin del JDK. Solución: verificar que se agregó %JAVA_HOME%\bin o $JAVA_HOME/bin al PATH, abrir una nueva ventana de terminal, y verificar ortografía (mayúsculas/minúsculas importan en Unix).

Problema 2: JAVA_HOME apunta a directorio incorrecto. Causa: ruta especificada incorrectamente. Solución: JAVA_HOME debe apuntar al directorio raíz del JDK sin /bin al final. Verificar con ls $JAVA_HOME (Unix) o dir %JAVA_HOME% (Windows) que muestra bin, lib, include, conf.

Problema 3: Múltiples versiones de Java, versión incorrecta ejecuta. Causa: otra instalación de Java está antes en PATH. Solución: asegurar que export PATH=$JAVA_HOME/bin:$PATH pone JAVA_HOME antes del PATH existente. En Windows, mover %JAVA_HOME%\bin al inicio de la lista.

Problema 4: Cambios no persisten después de reiniciar. En Windows: verificar que se usó "System variables" y que todos los diálogos se cerraron con OK. En macOS/Linux: verificar que se editó el archivo correcto (.bashrc, .zshrc, o .bash_profile) y que tiene permisos de lectura.

Configuración para múltiples versiones de JDK

Si se necesita cambiar entre múltiples versiones, se puede definir funciones en el shell para cambiar JAVA_HOME rápidamente:

  # En .bashrc o .zshrc

# Función para cambiar a Java 21
java21() {
    export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-21.jdk/Contents/Home
    export PATH=$JAVA_HOME/bin:$PATH
}

# Función para cambiar a Java 17
java17() {
    export JAVA_HOME=/Library/Java/JavaVirtualMachines/temurin-17.jdk/Contents/Home
    export PATH=$JAVA_HOME/bin:$PATH
}

# Uso: simplemente escribir java21 o java17 en terminal

Para gestión más avanzada de múltiples versiones, herramientas como SDKMAN y jEnv se cubren en el subtema 3.9.

La configuración correcta de JAVA_HOME y PATH es absolutamente fundamental. Todos los problemas que surgen posteriormente con herramientas de construcción, IDEs, o deployment frecuentemente trazan de vuelta a configuración incorrecta de estas variables. Invertir tiempo en configurarlas correctamente desde el inicio previene innumerables horas de frustración posterior.
